Zero Trust Model

  • Many networks do security checks very rarely once you are in the network or through the firewall, it makes authorized users esay to perform tasks, but what if someone outsider gain the access >?

  • Zero Trust is our go-to approach to perform additional check for every device, process, user, program, app whenever the request for resource is received.

  • Everything must be verified in the form of MFA, Encryption, System Permissions, Additional Firewall/Monitoring/Analytics, etc.

Applying Zero Trust Infrastructure

Planes of Operation

  • One of the way is to break our security device into several components and examining in each layer of splitted planes.

  • Either the security device is physical, virtual or in the cloud, you could essentially split into Data Plane and Control Plane.

  • Data Plane processes all incomming/outgoing frames, packets at the real time. It performs NAT, trunking, forwarding.

  • At the device, there is Control Place which is responsibe for actually consuming the data and passing it through the security check if it is legitimate or malicious traffic. You could write rules, policies, build routing tables, session tables, NAT tables to control data arrived at the Data Plane.

Controlling Trust

You should no longer believe or trust only what the user is providing. Of course, this is the first step of validation but you might consider several factors. This is actually the collection of information gathered about the user, process, device, etc. or essentially anything that should be taken as security check. Adaptive Identity is hot topic when it comes to controlling the trust based on many factors.

Adaptive Identity

  • Validation of source and requested resources,

  • Relation to the organization, physical location, type of connection, IP address, etc.

  • Make it as hard as possible, if you really want strong Zero Trust architecture.

  • Once we have analyzed these many variables, you can now verify the identity and provide access or the resource requested.

Another way in building Zero Trust foundation is :

Threat Scope Reduction

  • Reducing threat scope is descreasing possible entry points, yes this may be somewhat difficult you authorized person, but this could prevent unwanted party being in our network or protected zone.

Once we have all this information, we can now build :

Policy-driven Access Control

  • Combine these all variable to create a predefined set or rules for the security check.

  • You might have faced some error, if you were trying to change the password of your social media account from the newly login device. This is an example of Policy-driven Access Control where you are restricted to change password even if other variable you provided belongs to you (the real user).

If you are about to implement Zero Trust strictly in organization's internal, you could introduce the concept of :

Security Zones

  • Security is more than one-one relationship, yes User-1 might be authorized to access Resource-1, but only if he is coming from Trusted Zone.

  • Your job is to build these zones (Trusted/Untrusted, Internal/External, IT/HR/ACC) and apply relatively more or less security checks, create policies, internal rules based on the zones examining the Full Path of Communication.

  • You should identify the potential Policy Enforcement Point (PEP), the point where you need to place the policy for these additional variable checks. Choose the best point to implement the rules/policies.

  • PEP should gather all the info and pass to Policy Engine through Policy Administrator which is responsible for making decision related Grant, Deny or Revoke*.
