  • Cross Site Requests

    • Once you are logged in, you browser will trust the server and so do the server.

    • Say you are logged into facebook.com, now you have established trust to the server and your browser trust the request on your behalf

    • Now, what an attacker could do is create a malicious code to post on facebook from your facebook on their behalf.

    • They may email you some software or link, and if you click them the request is already made and as all happens on you->browser->server trust chain, and will ultimately be posted.

    • (Obviously, facebook will not allow direct access)

    • (Almost all web servers/websites have built in system to mitigate this attack.)

      • They will ask for the token for particular transaction

      • Even if you are trusted but obviously attacker does not have your token, unless they manage to gain access to your browser info, they can't create a code to make the request coz token is unknown

      • In the above example, the transaction is to post on facebook, before the use of tokenization technology, attacker will simply be able to write script or software or inject url to make the request, Now that the token is required, it is not that easy to perform task.

      • They could still manage to write code using brute force techniques, but that is hell of a strugle, upon these many unvalid request, browser may even block you
