Virtualization Vulnerabilities

  • Self contained system with its own OS, memory, storage, CPU, etc.

  • Runs at the top of Hypervisor software.

VM Escape

  • Simply means jumping from one VM to another within a single Hypervisor.

  • There could be 100s of VM in one hypervisor.

  • In general your Hypervisor provider or software itself should prevent VM Escape.

  • But bug inside Hypervisor could potentially lead to VM Escape and there have been some instances of this.

  • Update if any vulnerabilities found.

Resource Reuse

  • Your hypervisor software is responsible to manage resources for all of the VM running on that particular hypervisor.

  • Say you have 3 VMs, each allocated with 2GB of RAM, but in reality you have only 4GB on your host OS which the host and the hyperviosr will be sharing.

  • As 6GB is avl to hypervisor, it uses memory sharing concept i.e to use only when needed.

  • There could be the case where 2 VM tries to access the same memory location, and if you hypervisor ALLOW IT, it is the BUG.

  • This will eventually lead to data leak or even admin access to another VM.

  • For eg: VM2 may be able to read what VM1 writes.
