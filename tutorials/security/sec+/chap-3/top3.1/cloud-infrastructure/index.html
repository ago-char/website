Cloud Security

Refer to this video of professor messer : cloud infrastructure - Your could may be running as IasS, PaaS, SaaS, etc. - As cloud grows, the question arises who handles security ?

Security Metrix

  • If you are working with public cloud provider, they may provide a responsibility metrix.

  • It is overall description of who is responsible for what ?, this may include for eg :

    • You may be responsible for your cloud account
    • Cloud service provider may be responsible for updates in your account

  • Every cloud service provider may have different matrix and they may have well documented, or inform you before you persure their facility.

  • Matrix may look similar to this :

  • You can easily see that what is customer responsible for ? and what the provider is responsible for ?. You may want to implement security on where you are responsible.

Hybrid Cloud

  • The is the situation where you use more that one cloud (private/public) for your flexibility.

  • But with this flexibility, you may have to pay the price of additional responsibility, complexity and security.

  • These cloud may not talk to eachother. So, configuration and procedure of one cloud may not match to another. Also, merging your facility could be very difficult.

  • Firewall, server settings, logs writing may differ. And you may come into situation where you have to share and transfer data back and fourth from one provider to another. (This adds more risk of data security) So :

    • You may have to hire different experts for executing or even as security prespective.

Third Party Vendor in the Cloud

  • You may be working with the cloud service provider, but you also may have to work with third party service provider for the cloud.

  • They are like plug-ins in your browser.

  • While persuing the service of third party, you should have a policy to include them in :

    • Risk Assessment policy
    • Incident Response policy

  • You should also constantly monitoring the process and the avilability.

  • This is because for eg :

    • You put your server in your cloud.

    • But, you may wanna give firewall protection to the server with some third party provider who obviously partner with the cloud service provider.

    • You should be monitoring availability because, you can not expect a second without firewall, this could open security hole.

    • You may monitor process because you of course should be looking at latest version of software, data privacy and so on.

  • So, this is actually the chain between you, your cloud provider, and your third party provider. You should put everyone as part of process.

Infrastructure as code

  • Most cloud base system works as infrastructure as code.

  • This is require for configuring and defining the service you would like to choose and take.

  • You may write these code which is mostly in a text and your cloud service provider will deploy the service you wanted (server, network, database, application) on the basis of the code you provided.

  • This is easy because any time you wanna rebuild, the code defines your topology and infrastructure, this should be safeguard coz this may leak your configuration and open additional security hole in your cloud.

Serverless Architecture

  • When you think of server, you think of running some machine, configuring them to do certain task and you wil have to manage those each configuration.

  • But, cloud do also provide Serverless Architecture, where you do not have to worry what the backend or server is doing.

  • This is actually the deployment part that is done by the cloud itself.

  • You are just uploading your application or code.

  • This is done by Function as a Service (FaaS).

  • For eg : you write an application for sales department.

  • This may have image to process, customer details to process, sales data to process.

  • If you go with traditional architecture, you may have to run server for each of the process or at least run a big server to process these all. And they are running at once.

  • But in morden cloud way, when you want to accomplish certain task, your service provider will do all of the background task at run time.

  • For eg : you want to provide details of items sold with the image of item. Now you just ask for the data and the service provider :

    • Boot up function or create a function to fetch image
    • Another function for getting customer details
    • Another function for fetching price details and so on.
    • Like this, functions are containerized and referred to your Application.
    • Hence, function will automatically created and deployed at run time.
    • This will help reduce run time and reduce your cloud cost, also provide efficiency and scability.

  • See the following From Wikipedia > Serverless computing is "a cloud service category in which the customer can use different cloud capabilities types without the customer having to provision, deploy and manage either hardware or software resources, other than providing customer application code or providing customer data. Serverless computing represents a form of virtualized computing." according to ISO/IEC 22123-2.  1 (https://en.wikipedia.org/wiki/Serverless_computing#cite_note-:1-1) Serverless computing is a broad ecosystem that includes the cloud provider, Function as a Service, managed services, tools, frameworks, engineers, stakeholders, and other interconnected elements, according to Sheen Brisals

  • Your application developer is responsible for writing functions for these smaller part of application and upload. Rest of the task related deploying, hardware, structure, etc. is handled in the cloud itself.

Monolithic Applications

  • One big application which does every thing i.e.

    • User Interface
    • Business Logic
    • Data Input and Output

  • This will certainly increase the problem in :

    • Scalability (Deploy whole of application)
    • Security (Secure all the part of application with one logic)
    • Change control (each time when you update, you will have to reinstall all the code)

Microservices and API

AS #Monolithic Applications have certain drawbacks related to flexibility, scability or security, cloud server providers found solution in form of Microservices and API. You can see the model in the following picture :

  • As you can see, now every business logic is divided.

  • User will be provided as client management console.

  • Whenever certain tasks or functions are asked to be executed by client, it will make API call which will trigger certain microservice behind the scene.

  • While one service is performing, other services can simply sit down or perfrom the tasks asked. This certainly enhance efficiency.

  • So, now :

    • Security
    • Change Control
    • Management

  • These all are done at microservice level, so if you wanna apply security you can do it in pieces. For eg : when micro service is fetching image from bg, the security related backend be only applied. This certains helps in managing app and security.
