Cloud Vs On-Premises

  • Both comes with certain advantages and certain disadvantages.

Centralized Vs Decentralized

  • Most organizations are physically decentralized meaning that they might have :

    • Different location of services
    • Many cloud providers
    • Different OS running on different place

  • It becomes hard to mange them, because it requires security. And there should be the way for merging and managing them centrally.

  • As, the security can be centralized using centralized console, where :

    • You can manage all the networks, individual devices, etc.
    • Log and Analysis.
    • Notify and Alerting System.
    • System Status, Maintenance/patching, etc.

  • But this is NOT PERFECT, as this comes with the cost of :

    • Single point of failure
    • As org. grows, the information is overloaded
    • Hard to manage

  • You may want different decentralized management zone and for centralized zone, you should be able to access those decentralized zones in able to monitor the activity for each zone. This will certainly help. But as said, security is not perfect.

Virtualization

  • It alllow us to run different OS on our Host infrastructure.

  • You could create and tear them down completely and isolate from Host when needed.

  • But this has a problem, that for each VM, it requires specific OS even if the App running inside them is for same OS i.e App A, B and C all may be intended to windows but they need to be isolated. So for each App, you will spin up a new VM and this is obviously a headache.

Containers

  • Containers solves the problem of #Virtualization i.e. spinning same OS each time for different Apps to be run inside the VM by the following architecture :

  • Here, there is one OS on top our infrastructure (hardware), and we install docker (daemon) for container management. This maintains relation between theose Apps and our host OS.

  • This way we were able to achive same isolation regardless of spinning up same OS many times.

  • For security prespective, while using #Virtualization , you should have been worry about security of host, hypervisor, each os, each app. Now you should not worry about OS security each time, just focus on your host and app security.

IOT Devices

  • We may use different automation devices in out home or in org.

  • They may be :

    • Sensors (door lock, heating or colling devices)
    • Wearable (watches, health monitors)
    • Facility Automation (Tempereture, Air Quality, Lightiing)

  • They are connected to our network.

  • The problem is vendors are very good at manufacturing these devices and they will certainly enhance our usuage and efficiency but They may not be security Professionals.

  • There has been incidients that they stop providing support, stop providing patch version as they are task-driven not security-driven.

SCADA / ICS

  • SCADA (Supervisory Control and Data Acquisition System) is mostly used in large-scale industries. This is often referred as Industrial Control Systems (ICS).

  • This will take input manually or via sensors on Programmable Logic Controller (PLC) or Remote Terminal Unit (RTU) which then sends data to our server.

  • We now on Human Machine Interface (HMI) or SCADA Dashboard can view and filter the stats.

  • If you work in organization where there are tons of machinary, you will connect them in the given architecture using SCADA to manage and technicians will control all those machinery from central control room rather than physically visiting to every equipment installed. You can refer example to of :

    • Power Control Room
    • Oil or Water Supply
    • Even Traffic Control in Cities
    • Vehicle Monitoring

  • This system must be segmented from outside, this is applied to almost all of the organization who have SCADA. This tends to be the most secure system guarded because you can't imagine someoue outsider gaining access to Water Flow Control System for a big Amazon River. This will effect number of days or may arise national issues too.

Real Time Operating System (RTOS)

  • If you are using Windows, Linux or other OS, these are Non-Deterministic OS i.e. one process can't interfare another in normal uses.

  • But in some situation we need Deterministic OS, where we may have to focus entirely on one task or process. For eg :

  • If you are driving AutoDrive in your Car, and the system need to pull the break, the system at that point in time does not have luxuriy but to focus on one task i.e to break in and make you safe.

  • As in example you notice that they are too sensitive. These sensitive systems are used everywhere like militry, flight or others.

  • As we can't have a single security fault out there and actually in most case it is not easy to install anitivirus or firewall out in those system, we at least need to have similar kind of security posture.

  • And this is often accomplished via not giving access to OS as they are self contained sy stems. It is not that easy task to get into OS of those RTOS. Those are very restrictive. You could take example of simple location tracking chip of few mm (milimeter), there is not certain way to access that chip OS, as you will do with your Windows or Linux OS.

Embedded Systems

  • There are hardware and software binded together to perform specific function or to operate as a part of a very large system.

  • For eg : Your watch has specifc function.

  • Being specifc function (sole purpose) means those systems are very unlike to have centralised app store where you will download and upload those functions or apps. They are as it is having single purpose and they do that very well.

  • Common Examples :

    • Traffic Light Controllers
    • Digital Watches
    • Medical Imaging Systems

High Availability

  • One concern most security professionals have is to maintain UPTIME + AVAILABILITY of their equipments.

  • One of the way to achieve this is High Availability (HA). Thing to be noted is Redundany is not always availability because you can have redundant system but you may need time to install that system in Availiability Zone.

  • Even if one part of system failes, the system should constantly running, this is one step ahead of simply having Redundancy.

  • You can do this by following principle of primary-secondary i.e. if primary system fails for some reasons, secondary system will immediately will work in effect.

  • You could also make both the system running, so here both are primary and even one fails another will continue to work without even notice.

  • But is comes with the cost, for eg :

    • If we have 2 redundent servers running at the same time, it means the cost for equipment is 2X.
    • Along with it comes the cost of network, cost of power, cost of space.

  • So, due to high costing, you will eventually come at the point where you have to make business decision whether to buy another system or take the risk of that downtime
