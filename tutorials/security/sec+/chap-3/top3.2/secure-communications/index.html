How actually encryption happens

  • How are we able to encrypt all the data yet still find a way to send it across the internet hiding all of our information and send it across the vpn concentrator ?

  • A figure will help you clarify this :

  • The VPN concentrator often uses ipsec protocol to encrypt and decrypt traffic. Initially, the original traffic generated by application is now encrypted, as initial IP headers are encrypted, for routers to know where to reach, the IPsec will add new headers. When they reach destination, VPN concentrator will unwrap those Header and Trailer, also decrypt the encrypted part and forward to dedicated destination.

SSL/TLS VPN

  • This is one of the medium to connect back to the VPN Concentrator.

  • Uses ssl/tls, the same protocol to communicate and encrypt web sever traffifc, so generally bypassed by the firewall. This means no additional rules need to be defined at firewall level.

  • No big VPN clients or multi-clients are connected, generally single client connecting to the vpn provider.

  • Often software installed on mobile devices (nord vpn, hma etc., or propriteory level) that connect to vpn server.

  • Also integrated within os.

  • Normally authenticated with simple login username and password.

  • We are not forced to use shared passwords as in ipsec or some digital signs or certs, but we may have to capability to use them which depends on our vpn software provider and the security infrastructure.

  • Also VPN available for browsers, that will enc/dec traffic within browsers but not outside them.

  • SSL vpns can be set as always on, this will ensure that whenever we are sending traffic or receiving, that is always in the encrypted tunnel.

SD-WAN

Background

  • This is the old WAN architecture, where every remote branch or site will connect to the Data Center of the company, and then that Data Center will forward the traffic to provide the appropriate response that is asked by the site.

  • Today, with everything are gradually shifting toward cloud, and the topology becomes :

  • As, things are gone to the cloud, to get secure connection, we are now required to connect to the data center, and data center will then connect back to the cloud and gets the information to provide us.

  • There is loss in network bandwidth, jitter, latency, etc. with this approach. Because to get the resource availabe at the cloud, we are (unnecessarily) visiting our data center for security.

SD-WAN Architecture

  • sd wan designes network in such a way that, it will connect back to data center only when necessary. Different rules, paths, ACLs are deployed to achieve this.

Secure Access Service Edge

  • Now that we have a way to connect to cloud and data center in network efficient way, thanks to SD-WAN, How will me maintain the secure Infrastructure ?

  • We will now use SASE.

  • Alongwith the services are shifted to cloud, security is too. And the security will be very close or next to the services we are trying to render.

  • We can think SASE as our Next Generation VPN with a lot of capabilities like :

  • Now that each users will be configured to use SASE, they will first connect to SASE which is capable of providing all the necessary functions.

  • After connection to SASE, user then can jump over the services required.
