Backups

Keeping backups for data is extremely important for any organization as it is the only plan for recovery or disaster. But several variables should be considered before perfoming backups :

Total Amount of Data

  • You should know what amout of data should be backup i.e. if it is all the data in disk or array, specific files, or filesystem or directory.

  • This result in how much storage capacity (will it be on GBs or TBs or more) to be bought and so on.

Backup Media

  • You should also choose backup media if to be SSD, HDD or else depending on your data type or requirement or budget.

Frequency of Performing backup

  • You should understand how often you should keep backup.

  • If you are non tech individual and data in your device does not change much, you could have long duration. It changes, when you are techhie.

  • But if you are company, where data flows in real time, you should perform backup maybe every day, every hour, etc.

Type of Backup

Snapshots

  • With the advent of VM, it has been common to use snapshots as backups where only the updated part of the backup will be stored. This will save storage as only that information to revert changes will be stored.

  • As seen, if you wanna revert back to Monday Backup, you are just a click away but behind the scene, those additional block added will be reverted.

Replication

  • Unlike snapshots, everything on original or primary system will be copied to redundant or backup.

  • This often happens on real time, once you write data in primary system, it will be automatically copied over backup in matter of seconds or minutes.

  • This is extremely useful while creating "hot site".

Journaling

  • Journaling is a special kind of backup.

  • What if power cuts between writing data from application to block ?

  • After you turn on power or power comes, the data may only be written few blocks, but in many cases the data will be corrupted and you may have to flush entire database which will certainly be data loss.

  • Your company can't afford data loss.

  • So, many databases, filesystems, OS implicitly have methods of journaling.

  • First the data will be written in journal file, only then after that it will be written in the disk.

  • If journal is corrupted, it does not affect the database, as it has never be written.

  • But what if power cuts between writing in disk ?

  • You can now, resume from where left or delete corrupted part and then wirte from journal again.

  • You can see those journal files in database, bind9 server, etc.

Storage Location

  • Storage location is significant in designing backup plan.

  • It might be on site or off site or in cloud.

  • It depends on amount or type of data you are trying to backup.

  • If it is shared, you may want to backup in cloud, if it is private you may want on-site, if it is in bluk, you may wanna transfer in different site.

Backup Software

  • You should choose faster, smooth and easier to use software.

  • For eg : if you are individual and performing less amout to data you can use simple cp command, but again if you have multiple files, it will be slow using cp, you may try rsync.

  • OS, themselves will have the facility to backup and restore to some extent, but if you are entreprise and wanna have more control on how and what you backup, you should try other third party softwarers like : Veeam Backup and Replication.

Encryption

  • Either you are transporting your data in physical storage to different data center or even keeping in your own or pushing to cloud or given to third party, there is always risk of data stolen.

  • Even if someone were able to access, they should not be getting anything out of it.

  • So, you are responsible for encryption of your data.

  • But then, you should keeps your keys safe and implement access control.

Recovery Testing

  • Even if you perform backup, you should test that if you are able to recover if something happens.

  • After backup, you should test everything runs as before.
