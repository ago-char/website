Recovery Testing

You may have defined your way for resilience and plan your capacity, but you also need to test if they will work if something were to happen.

It's not "if" happens, it's often "when" happens

  • Testing should be done in confined manner.

  • None of testing activity should affect any kind of production activities.

  • Often, extra time or off day training is organized for this.

Tabletop Exercise

  • It is almost impossible to shilf every employee, infrastructure to shift in testing environment specially in large organizations.

  • As, it is hard to create exact recovery state, tabletop exercise, is nice approach for at least giving all the stakeholders idea and knowledge about it.

  • In the exercise, it is assumed that situation has arrived, individuals will be tasked to perform in specific time and analysis the result.

  • The best way of perfoming this exercise, is to talk through it or organize virtual presentation class.

Simulation

  • You should simulate the exact training cross check.

  • You may have provide training on phising, so to cross check, org should conduct this program of sending phising campaign invisible to employees and evaluate the result.

  • If to check the system can recover from fail over, you should perform exercise like turning off the original server and check if it could re catch the secondary one. This should be done in testing, outside the production environment.

Parallel Processing

  • This may not be the best way but using multiple core or multiple cpus for parallel processing also handels recovery in certain way.

  • If a core of cpu fails, still some other cores will be able to perform intended task, even slowness occur, something is better than nothing.
