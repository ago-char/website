OVERLAYS

Overlays are software components that provide hooks to functions analogous to those provided by backends, which can be stacked on top of the backend calls and as callbacks on top of backend responses to alter their behavior.

Overlays may be compiled statically into slapd, or when module support is enabled, they may be dynamically loaded. Most of the overlays are only allowed to be configured on individual databases.

Essentially, overlays represent a means to:

  • customize the behavior of existing backends without changing the backend code and without requiring one to write a new custom backend with complete functionality
  • write functionality of general usefulness that can be applied to different backend types

Think of this as extention in browsers. Say, you want to log every read and write activity done in certain database. One way to achieve this is to write backend, that will do read, write, and also log in special database. This is tedious, because LDAP already and perform read, write, etc. To log is put overlay somewhere in between frontend and backend component, so the overlay software can log the activity even before reaching to the backend. This is flexible because as user POV, he is doing what he has been doing, but you as an admin now have the overview somewhere dumped which later can be analysed. Pictorically :

Overlays are usually documented by separate specific man pages in section 5; the naming convention is : slapo-<overlay name>

If you are interested in what overlays are available, you can check man page. Typically, following are the steps to make an overlay into work :

  1. Overlays are used as modules, these are available in your library, say in Debian /var/lib/ldap. So, import any overlay you wanna use as module.
  2. Create Database or Branch in the DIT (Directory Information Tree) for the module to work its internal functionalities.
  3. Add Overlay to required database usually under cn=config.
